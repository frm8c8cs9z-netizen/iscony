# Specification Draft

このメモは、`iscony` の仕様を会話ベースで少しずつ積み上げるための下書きです。  
実装済みの事実と、今後の方針を混ぜずに残していくための受け皿にします。

## 1. 用語と役割

- `iscony` は本システムの名称。
- `Stage` は内部実装上の概念で、運営者には前面に出しすぎない。
- 運営者は「どこの結果がどこの枠に入るか」「どの試合をどう進めるか」を主に意識する。
- 一般利用者は、カテゴリ結果、進行表、トーナメント表を流れで見る。

## 2. 当日運営の入口

当日運営の入口は複数あってよい。

- `試合選択`
  - 進行枠、コート、試合順を選んで素早く対象試合へ入る入口。
  - 現場で押しやすい軽い導線として扱う。
- `試合受付・結果入力`
  - 既存の検索導線をまとめた結果入力ハブ。
  - 現時点では `カテゴリ + 番号` と `コート + 第何試合` の検索を残す。
  - `match_key` の直接入力検索は第1段階として実装済み。
  - QR 読み取りは試合受付の補助入口として追加し、読めない端末では手入力へ落とす。
- どの入口を先に出すかは、端末や受付の状況で変わりうる。

## 3. 結果入力ハブの考え方

- 結果入力は、最終的にはモーダル化したい。
- モーダルには、カテゴリ、Stage、リーグ/トーナメント名、コート、試合順、マッチ入力先などの文脈情報をまとめて出す。
- リーグ表やトーナメント表からも同じモーダルへ入れるようにする。
- 未確定の情報は空欄でもよいが、どこが未確定か分かるようにする。

## 4. 試合識別キー

将来的に、スコアシートや QR から試合を直接特定できる `match_key` を持たせる。

### 方針

- 外に出すキーは DB の `id` をそのまま見せない。
- 進行やコートが変わっても、同じ試合を指し続けられるようにする。
- 数字だけで入力しやすい形を優先する。

### 現時点の合意

- 種別は数字で表す。
  - `1` = リーグ
  - `2` = トーナメント
- 本体番号は可変長の数字にする。
- ゼロ埋めは入力前提にしない。
- チェックデジットは Damm 法を第一候補とする。
- バーコードは QR とは別扱いで、まずは QR を前提にする。

### 目安

- 例: `1-12-7`
- 例: `2-245-4`

ここでの区切り記号は、表示や印刷の都合で変えてよい。

## 5. 公開画面と管理画面

- 公開画面は参加者・関係者が見るもの。
- 管理画面は当日運営者・準備担当者が触るもの。
- 公開画面には、結果入力や進行変更などの運営操作を出さない。
- 管理画面には、必要に応じて試合選択、結果入力、採点票、進行変更の入口を残す。
